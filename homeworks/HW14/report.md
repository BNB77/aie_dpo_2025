# HW14 – эмбеддинги, FAISS, оценка retrieval и mini-RAG по базе знаний

## 1. Кратко: что сделано

- **База знаний:** 25 коротких статей-определений по алгоритмам и методам ML/AI (inline в ноутбуке, воспроизводимо без внешних загрузок). Выбрана как тематически связная область с очевидными предметными запросами.
- **Чанкинг:** посимвольный, `chunk_size=300`, `overlap=50`; получено ~60 чанков. Каждый чанк содержит 2–3 связных предложения — достаточно для точного поиска.
- **Embedding-модель:** `all-MiniLM-L6-v2` (sentence-transformers), 384-мерные векторы, косинусное сходство.
- **Оценка retrieval:** 10 контрольных запросов, `hit@5 = 1.0`, `recall@5 = 1.0` — модель уверенно находит нужный документ по всем запросам.
- **Обновление БЗ:** добавлено 5 новых документов (Batch Normalization, ResNet, BERT, Feature Importance, Ensemble Methods); после переиндексации 5 из 6 тестовых запросов изменили топ результатов в пользу новых документов.
- **Mini-RAG:** extractive pipeline (retrieve → context → первые 2 предложения лучшего чанка → ответ с источниками). На точечных запросах работает хорошо; сравнительные запросы остаются проблемными для extractive-подхода.

## 3. База знаний и подготовка документов

- **Предметная область:** алгоритмы и методы машинного обучения (Linear Regression, Neural Networks, Transformers, Clustering, и т.д.).
- **Источник документов:** inline-данные в ноутбуке (написаны вручную, ~200–400 символов каждый).
- **Число исходных документов:** 25; после обновления — 30.
- **Число чанков после разбиения:** ~60 (базовый набор), ~73 (обновлённый).
- **Способ чанкинга:** посимвольный с фиксированным размером и перекрытием.
- **Параметры:** `chunk_size=300`, `overlap=50`.
- **Комментарий:** ML/AI концепции удобны для retrieval-задачи, так как каждый документ посвящён одной чётко определённой теме. Запросы вида «как работает X» или «в чём отличие Y от Z» имеют однозначный ожидаемый источник. Размер chunk_size=300 охватывает 2–3 предложения, что соответствует одной законченной мысли. Overlap=50 снижает риск разрыва связного контекста на границах чанков. База достаточно мала для быстрого учебного запуска, но достаточно разнообразна, чтобы retrieval имел смысл.

## 4. Retrieval и индекс

### 4.1. Эмбеддинги и индекс

- **Embedding-модель:** `all-MiniLM-L6-v2` (sentence-transformers), 384-мерные векторы.
- **Метрика сходства:** косинусное сходство (векторы нормализованы L2, индекс `IndexFlatIP`).
- **Индекс FAISS:** `IndexFlatIP` — точный перебор по внутреннему произведению.
- **top_k по умолчанию:** 5.
- **Первые примеры retrieval:** модель уверенно находит тематически близкие документы; запрос про backpropagation возвращает doc_10 на 1-м месте, запрос про k-means — doc_22 на 1-м месте.

### 4.2. Контрольные запросы и оценка retrieval

- **Контрольных запросов:** 10.
- **Релевантным** считался документ-источник с соответствующей концепцией (1–2 документа на запрос).
- **Метрики:** `hit@k` и `recall@k` при k=5.
- **hit@5:** 1.000 (10/10 запросов).
- **recall@5:** 1.000.
- **Простые запросы:** K-Means, Dropout, PCA, Backpropagation — нашлись на 1-м месте.
- **Потенциально трудные:** запрос по Transformer/Attention ожидал два документа (doc_14, doc_15) — оба найдены в top-5.

## 5. Эксперимент по параметрам retrieval и обновление базы знаний

- **Сравниваемый параметр:** `chunk_size`.
- **Варианты:** 200 vs 300 vs 400 символов (overlap=50 фиксирован).
- **Результаты:** при chunk_size=200 и 400 hit@5 и recall@5 аналогичны 300 на данной базе. chunk_size=200 даёт больше коротких чанков (риск неполного охвата мысли); chunk_size=400 — меньше чанков с более длинными фрагментами (снижение точности косинусного сходства).
- **Выбран chunk_size=300** как баланс между охватом контекста и точностью поиска.
- **Обновление БЗ:** добавлены 5 новых документов (Batch Normalization, ResNet and Skip Connections, BERT Language Model, Feature Importance, Ensemble Methods).
- **Переиндексация:** повторный вызов `chunk_documents` + `model.encode` + `IndexFlatIP.add`.
- **Изменения в retrieval:** 5 из 6 тестовых запросов получили новые документы на 1-м месте вместо приближённых аналогов. Запрос про Dropout не изменился — его источник (doc_19) уже присутствовал в базе.

## 7. Анализ

Выбранная база знаний оказалась удачной для учебной задачи: каждый документ посвящён одному понятию, что обеспечивает чёткую связь запрос–ответ и упрощает оценку hit@k. Посимвольный чанкинг с размером 300 символов хорошо разбивает короткие статьи на 1–2 чанка, сохраняя связность предложений. Использование `all-MiniLM-L6-v2` дало хорошее качество — модель семантически понимает запросы и сопоставляет их с нужными документами даже при разных формулировках.

Формальная оценка retrieval (hit@k, recall@k) подтвердила качественное впечатление: все 10 запросов находят нужный документ в top-5. Эксперимент с chunk_size показал, что при данном масштабе базы (25 документов) выбор размера не критичен, но при большей базе разница была бы более заметной.

Обновление базы знаний продемонстрировало ключевое свойство retrieval-архитектуры: переиндексация с новыми документами немедленно улучшает поиск по ранее неохваченным темам. Основное ограничение текущего решения — extractive-генератор: он не синтезирует информацию из нескольких источников, что делает его неподходящим для сравнительных запросов. Замена на генеративный LLM исправила бы этот недостаток при сохранении retrieval-части без изменений.

---

## 8. Итоговый вывод

Базовым retrieval-конвейером для такой задачи остаётся: sentence-transformers (all-MiniLM-L6-v2) + посимвольный чанкинг 300/50 + FAISS IndexFlatIP. Главное про качество retrieval: при чётко структурированной базе знаний с тематически однозначными документами hit@5 достигает 1.0 даже с компактной моделью. Главное про связь retrieval и mini-RAG: ошибки retrieval напрямую определяют ошибки RAG — если нужный документ не найден, ответ будет принципиально неверным независимо от качества генератора; поэтому инвестиции в retrieval важнее, чем в генерацию, особенно для небольших баз знаний.
